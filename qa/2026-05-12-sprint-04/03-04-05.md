## Findings

- FAIL ‚Äî **Story 03 AC-1: Port method name mismatch.** The story's acceptance criterion says the port must expose `topN(n)`. The implementation method is named `top(n)` throughout (`score-store.js:92, 130`). The HTTP handler and ADR also use `top`. The AC text is the contract; the implementation does not satisfy it by name. Callers who depend on the story's stated interface (`topN`) would fail.

- FAIL ‚Äî **Story 03 AC-3: No fsync.** The AC states "`award` ... fsyncs before resolving." `score-store.js:82` calls `fs.appendFileSync` which does not issue an fsync syscall ‚Äî it writes synchronously to the kernel buffer only. No `fs.fsyncSync(fd)` call exists anywhere in the store. The JSDoc comment on line 65 says "fsync" but the code does not do it. This is a documented requirement that is not implemented.

- FAIL ‚Äî **No integration test for the end-to-end win ‚Üí award ‚Üí leaderboard flow.** The integration suite (`test/integration.js`) never plays a full game through the WebSocket to produce a winner, then verifies `GET /api/leaderboard` shows the winner with +1 pt. The leaderboard integration tests (lines 432‚Äì471) register a fresh user, call `GET /api/leaderboard`, and assert shape/auth/cap ‚Äî but the store is empty for all of these calls. As a result, `data/scores.jsonl` is 0 bytes after a full `./verify.sh` run. The scenario specified in the QA brief (Alice and Bob play a winning sequence over WS; GET /api/leaderboard then shows Alice with +1 pt) is absent. This means the integration between MatchHub's `award` call and the real `JsonlScoreStore` is never exercised end-to-end.

- WARN ‚Äî **`data/scores.jsonl` is 0 bytes after full verify run.** Because no integration test drives a win through the live server, the production adapter (`JsonlScoreStore`) is never exercised during `./verify.sh`. The JSONL persistence guarantee (AC-3 story 03, "persists across a server restart" scenario) is untested at the integration level.

- WARN ‚Äî **Story 03 AC-3 / AC-5 NFR gap: `award` call order relative to broadcast.** `match-hub.js:95‚Äì98` calls `award(...).catch(...)` then immediately calls `_broadcastState` and `_broadcast`. Because `award` is async-returning (even though its body is synchronous), the Promise resolves in a microtask. In practice the in-memory map is updated before the broadcast (body is synchronous), but the JSONL write also completes before the microtask returns, so timing is fine. This is a documentation gap, not a functional bug, but it could mislead future maintainers.

- PASS ‚Äî `server/index.js` wires `scoreStore` into `MatchHub` correctly (line 29).
- PASS ‚Äî `InMemoryScoreStore` is used in all unit tests; no disk I/O in `test.js`.
- PASS ‚Äî draw and abandon paths confirmed no-op for scoring (unit tests + code).
- PASS ‚Äî `leaderboard.html` uses semantic `<table>` with `<caption>`, `scope` headers, `tabindex="0"` on rows, `aria-live="polite"`.
- PASS ‚Äî empty-state message present: "No scores yet ‚Äî play a match to get on the board!"
- PASS ‚Äî error-state message present: "Could not load scores ‚Äî please try again."
- PASS ‚Äî "Back to lobby" link visible and functional in all rendering paths.

## Suggested Fixes (RED items)

1. **Story 03 AC-1 ‚Äî rename `top` to `topN`** (or update AC-1 to say `top`). The port contract as stated uses `topN`; the implementation uses `top`. One must be changed to match. Given the ADR and all callers use `top`, the faster fix is to update AC-1 text; if the story's naming is authoritative then rename the method everywhere and update `server/index.js:110` and the ADR.

2. **Story 03 AC-3 ‚Äî add `fs.fsyncSync`** after `appendFileSync` in `JsonlScoreStore.award`. The synchronous write path makes this straightforward:
   ```js
   const fd = fs.openSync(this._file, 'a');
   fs.writeSync(fd, line);
   fs.fsyncSync(fd);
   fs.closeSync(fd);
   ```
   Or open the file once in the constructor with `fs.openSync` and fsync on each award.

3. **Missing integration test ‚Äî add an end-to-end win‚Üíaward‚Üíleaderboard scenario** to `test/integration.js`. The test should:
   - Register and login alice and bob as fresh users.
   - Alice creates a match; bob joins.
   - Both connect WebSocket and subscribe.
   - Alice plays cells 0, 1, 2 (with bob playing 3, 4 in between) to win top row.
   - After receiving `match.ended` with `winner` set, call `GET /api/leaderboard` with alice's session cookie.
   - Assert the response array contains `{ username: alice, pts: 1 }`.
   This is currently the only path that exercises `JsonlScoreStore.award` in a running server context.
